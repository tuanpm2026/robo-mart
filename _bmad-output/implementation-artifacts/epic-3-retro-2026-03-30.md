## What Went Well

### 1. Security Architecture — Framework-Agnostic security-lib
The decision to create `security-lib` as a framework-agnostic module (no Spring Boot starters) was key. It works with both servlet-based services (product, cart) and reactive WebFlux gateway. `KeycloakRealmRoleConverter` and `AuthContext` are reusable across all services without dependency conflicts.

### 2. OIDC Standard Compliance — oidc-client-ts over keycloak-js
Choosing `oidc-client-ts` over the Keycloak-specific JS adapter was the right call. It provides standard OIDC support, lighter bundle size, and better TypeScript types. The silent renew via hidden iframe and session storage token strategy provide good security defaults.

### 3. Non-Blocking Cart Merge Pattern
Story 3.4's merge design is resilient by default: merge failure never blocks login, `initAuth()` provides a fallback if callback merge fails, localStorage anonymous ID is only cleared on success, and concurrent merge calls are idempotent. This "fire-and-forget with retry" pattern is a good model for future cross-cutting concerns.

### 4. API Gateway as Central Security Enforcement
All security enforcement happens at the gateway level — RBAC rules, JWT validation, user identity relay. Backend services trust `X-User-Id` from the gateway without re-validating JWT. This centralized model simplifies service-level security and enables consistent policy changes.

### 5. Progressive RBAC with Placeholder Routes
Story 3.3 added routes for services that don't exist yet (order-service, admin-products, admin-orders). This validates RBAC rules early and means Epic 4 services will be secured from day one. Smart forward-thinking without over-engineering.

## Challenges & Struggles

### 1. Spring Cloud Gateway is Reactive (WebFlux)
Spring Cloud Gateway uses WebFlux, which means servlet-based security APIs (SecurityFilterChain with HttpSecurity) don't work. Required separate reactive security config (`ServerHttpSecurity`). This also means `security-lib` can't import any servlet or reactive starters — it must remain agnostic.

### 2. Spring Boot 4 OAuth2 Starter Renames
OAuth2 resource server starter was renamed from `spring-boot-starter-oauth2-resource-server` to `spring-boot-starter-security-oauth2-resource-server` in Spring Boot 4. Gateway starter also renamed from `spring-cloud-starter-gateway` to `spring-cloud-starter-gateway-server-webflux`. These renames are poorly documented and required trial-and-error.

### 3. WebTestClient Not Auto-Configured
In Spring Boot 4, `WebTestClient` is no longer auto-configured in `@SpringBootTest`. Required manual `WebTestClient.bindToServer()` with `@LocalServerPort`. Combined with `@MockitoBean` replacing `@MockBean`, test setup patterns are different from Spring Boot 3 examples.

### 4. CORS + Keycloak Redirects
CORS configuration for Keycloak OIDC redirects required `add-to-simple-url-handler-mapping: true` in gateway config, plus `DedupeResponseHeader` filter to prevent duplicate CORS headers. Non-obvious configuration that's easy to get wrong.

### 5. Vitest Mock Hoisting
`vi.mock()` factory functions are hoisted to the top of the file — they can't reference `const` variables defined outside. Mock data must be inlined inside the factory, which makes tests more verbose but avoids subtle runtime errors.

## Key Insights & Patterns

### Technical Patterns Established

1. **Framework-Agnostic Security Module**: `security-lib` with no Spring Boot starters — works with servlet and reactive consumers
2. **Keycloak Realm Role Converter**: Custom `GrantedAuthoritiesConverter` for nested `realm_access.roles` JWT claim
3. **UserIdRelayFilter (Gateway)**: Central identity management — JWT sub for authenticated, validated header for anonymous
4. **OIDC Silent Renew**: Hidden iframe pattern with `automaticSilentRenew` for seamless token refresh
5. **Non-Blocking Cart Merge**: Fire-and-forget with fallback — merge failure never breaks login flow
6. **Progressive RBAC**: Add security rules for future services early; validate before implementation
7. **Anonymous→Authenticated Transition**: localStorage UUID → merge → clear pattern for identity handoff

### Process Insights

- **Security-first works**: Adding security infrastructure early (Epic 3 before business logic in Epic 4) avoids retrofitting auth into existing services
- **Gateway simplifies services**: Backend services don't need to handle JWT — they trust X-User-Id from gateway
- **Placeholder routes reduce integration risk**: RBAC rules tested before target services exist
- **Non-blocking merge is the right default**: Login UX must never be degraded by secondary operations

### Architecture Decisions Validated

- Spring Cloud Gateway as centralized security enforcement point — correct choice
- Keycloak as external IdP (not custom auth) — saves massive effort
- `oidc-client-ts` over `keycloak-js` — standard OIDC, lighter bundle, better TS support
- `sessionStorage` for tokens — tab isolation, auto-cleared on close, better than localStorage
- Framework-agnostic `security-lib` — proven reusable across servlet and reactive services

## Key Takeaways

1. **Security-first pays off** — Adding auth infrastructure in Epic 3 (before business logic in Epic 4) means all future services are secured from day one. Retrofitting auth is always harder.
2. **Framework-agnostic shared libraries are essential** — `security-lib` works across servlet and reactive services because it avoids Spring Boot starters. This pattern should be replicated for any cross-cutting concern.
3. **Non-blocking secondary operations improve UX** — Cart merge on login is fire-and-forget with fallback. Login never fails due to merge errors. Apply this pattern to future cross-cutting concerns (notifications, analytics, etc.).
4. **Gateway centralization simplifies services** — Backend services don't handle JWT or RBAC — they trust the gateway. This reduces per-service security complexity and enables consistent policy changes.
5. **Placeholder routes enable early validation** — Adding RBAC rules for future services (orders, admin) during Story 3.3 caught configuration issues early and means Epic 4 integration will be smoother.
6. **Deferred work is trending down** — Epic 3 added only 6 deferred items vs Epic 2's 19. Most are pre-existing decisions, not new debt. The codebase is stabilizing.
