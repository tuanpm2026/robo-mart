## What Went Well

### 1. gRPC Research Upfront — Biggest Risk Neutralized Early

The decision to research `spring-grpc` + Spring Boot 4 compatibility before Story 4.1 (Action Item from Epic 3) eliminated the biggest unknown in the epic. `spring-grpc` 1.0.2 worked correctly with `@GrpcService`, `GrpcChannelFactory`, and `GrpcServerLifecycle.getPort()` — all documented in MEMORY.md before any saga code was written. Story 4.4 (the most complex) was not blocked by infrastructure uncertainty.

### 2. TransactionTemplate Pattern — Discovered Once, Reused Everywhere

Story 4.2 discovered that `@Transactional` doesn't compose well when wrapping external calls (Redis distributed lock must wrap the DB transaction, not the reverse). The `TransactionTemplate` programmatic pattern was adopted and reused verbatim in Stories 4.3, 4.4, and 4.5 without re-discovery.

### 3. Saga Built Incrementally — Complexity Well-Managed

Story ordering was intentional: 4.1 (infrastructure) → 4.2 (Inventory gRPC) → 4.3 (Payment gRPC) → 4.4 (Saga orchestrator) → 4.5 (Cancellation). Each story built on stable, tested foundations. Story 4.4 could implement the full saga because gRPC services were already proven.

### 4. Dev Notes Knowledge Accumulation

The "Previous Story Intelligence" section in each story file prevented re-discovery of the same issues. Jackson 3.x imports, `GrpcServerLifecycle.getPort()`, `TransactionTemplate` pattern, `OutboxEvent` constructor — each discovered once, then carried forward as explicit guidance.

### 5. Code Review Caught Production-Grade Bugs

Several review findings were serious defects, not style issues:
- **Story 4.3**: FAILED payment stored idempotency key → retry permanently blocked by unique constraint
- **Story 4.4**: `kafkaTemplate.send()` is async — crash after send but before mark-published causes duplicate events
- **Story 4.5**: **Auth bypass** — any authenticated user could cancel any other user's order

Without the 3-layer review process, at least the auth bypass would have shipped to production.

## Challenges & Struggles

### 1. gRPC Testing — `@LocalGrpcPort` Removed in Spring gRPC 1.0.2

`@LocalGrpcPort` annotation does not exist in Spring gRPC 1.0.2. Required `GrpcServerLifecycle.getPort()` instead — discovered by trial-and-error in Story 4.2 and propagated to all subsequent stories manually.

### 2. Saga Recovery Edge Cases — More Than Expected

Story 4.4 required 5 review patches on saga recovery and state management alone:
- Lazy-loaded items collection empty during `recoverStaleSagas()` → empty `ReleaseInventoryRequest`
- Incorrect `previousStatus` passed to `publishStatusChangedEvent()` during recovery
- `getOrder()` outside transaction → `LazyInitializationException` when iterating items
- `createOrder()` leaves order stuck in PENDING if `executeSaga()` throws unexpected exception

Distributed systems recovery logic has more edge cases than synchronous business logic.

### 3. Outbox Polling — At-Least-Once Delivery Harder Than Expected

`kafkaTemplate.send()` is async (returns `ListenableFuture`). If the service crashes after send but before the DB `mark-published`, the event is published twice on next poll. Fix: call `.get()` for synchronous confirmation. This is a production-grade race condition that was invisible in happy-path testing.

### 4. Payment Idempotency — Three Concurrent Issues Found Together

Story 4.3 review found three related race conditions simultaneously: TOCTOU on idempotency key check (two threads both pass check), retry blocked after transient failure (FAILED payment locks idempotency key), and non-atomic expired key deletion. Root cause: concurrent access scenarios not covered in initial design.

### 5. Frontend Checkout Complexity

Story 4.7 required coordinating: VeeValidate + Yup validation, 4-step PrimeVue Stepper, SagaProgressOverlay, error routing between steps (PAYMENT_FAILED stays on step 3, INVENTORY_FAILED redirects to /cart), and synchronous backend saga call. The most complex frontend story in the project to date.

## Key Insights & Patterns

### Technical Patterns Established

1. **TransactionTemplate over @Transactional** — When transaction must wrap external calls (Redis lock, gRPC), use programmatic `TransactionTemplate`. Lock acquires outside, transaction inside.
2. **gRPC Error Code Contract** — `FAILED_PRECONDITION` = permanent (compensate immediately), `UNAVAILABLE` = transient (retry with backoff), `NOT_FOUND` = missing resource (skip gracefully)
3. **Best-Effort Saga Compensation** — For cancellation flows: refund failure logs and continues, never leaves order stuck in intermediate state. Manual reconciliation is the fallback.
4. **Outbox Polling Delivery** — `kafkaTemplate.send().get()` for synchronous confirmation. Without `.get()`, at-most-once semantics on crash.
5. **`GrpcServerLifecycle.getPort()`** — Standard pattern for gRPC port discovery in integration tests. `@LocalGrpcPort` does not exist in Spring gRPC 1.0.2.
6. **SagaContext as POJO** — Steps mutate Order entity directly via context reference. No intermediate DTO needed.
7. **Lazy Loading Guard** — Any gRPC/service method that iterates JPA collections must either JOIN FETCH or use `@EntityGraph`, or wrap in `TransactionTemplate`.

### Process Insights

- **Infrastructure story before service stories pays off**: Story 4.1 investment meant Stories 4.2-4.5 were never blocked by infrastructure unknowns
- **Story ordering matters for distributed systems**: gRPC contract stability (4.1-4.3) is a prerequisite for Saga design (4.4) to avoid refactor loops
- **Code review must include concurrent scenarios**: Every serious bug in this epic was a concurrency/race condition, not a logic error

### Architecture Decisions Validated

- `spring-grpc` 1.0.2 as gRPC integration layer — works with Spring Boot 4.0.4, auto-configures server and client channels
- Per-SKU Redis lock granularity for inventory — correct choice, allows parallel reservations for different products
- Saga orchestrator pattern (not choreography) — explicit state machine makes compensation paths clear and testable
- Best-effort compensation for cancellation — pragmatic trade-off: order consistency over perfect financial accuracy at service boundaries

## Key Takeaways

1. **Research before complexity pays dividends** — The gRPC research before 4.1 meant zero infrastructure surprises during the most complex epic. Apply to any new pattern entering the codebase.
2. **TransactionTemplate is the right default for distributed systems** — Anywhere external calls mix with DB operations, programmatic transactions win over declarative `@Transactional`.
3. **Code review must actively look for concurrent scenarios** — Every serious bug was a race condition invisible to single-threaded happy-path tests. Add concurrent scenario review as an explicit checklist item.
4. **Auth checks are a first-class concern, not an afterthought** — The auth bypass in Story 4.5 was found only in review. Every write endpoint must explicitly validate user ownership.
5. **Best-effort compensation is a valid production pattern** — Not every failure needs full ACID consistency. Document the trade-off, log the failure, enable reconciliation. Don't let perfect be the enemy of good.
6. **Deferred work trending upward** — 60 total items. Epic 8 (Resilience) will be critical to address distributed locking, timeouts, and race conditions before production readiness.
