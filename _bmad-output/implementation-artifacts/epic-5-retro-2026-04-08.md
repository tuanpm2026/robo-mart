## What Went Well

### 1. Fastest Epic Velocity — 5 Stories in 3 Days

Epic 5 was the fastest delivery in the project. Strong pattern reuse from previous epics — admin controller convention, Pinia store template, DataTable patterns — allowed stories 5.4 and 5.5 to be near-copy-paste in structure.

### 2. Admin Dashboard Bootstrapped Cleanly

First time running 2 frontend apps in npm workspaces (customer-website + admin-dashboard). `@robo-mart/shared` with `adminTheme` worked on first attempt. No workspace resolution issues. Workspace linking pattern proven at scale.

### 3. Code Review Caught CVE-Level Security Bugs

Story 5.3 review discovered:
- **Path traversal** in `ImageStorageService.delete()` — attacker could read arbitrary files
- **Content-type spoofing** — client sends `image/jpeg` header but file is executable; fixed with magic bytes detection
- **TOCTOU race** on image count limit — fixed with `SERIALIZABLE` isolation
- **Orphaned files** when DB save fails mid-loop — fixed with cleanup in catch block

Without review, path traversal would have shipped to production.

### 4. Consistent Admin Controller Convention

Established in Story 5.2, followed verbatim in 5.4 and 5.5:
- No `@PreAuthorize` — gateway-enforced RBAC
- `Tracer` injection for trace ID in responses
- `@ExceptionHandler` for `ObjectOptimisticLockingFailureException` → 409
- Controller in `com.robomart.*.controller` package (separate from customer-facing `web` package)

### 5. Story Ordering Prevented Blocking Dependencies

5.1 (foundation) → 5.2 (CRUD, establishes patterns) → 5.3 (images, builds on CRUD) → 5.4 (inventory, separate service) → 5.5 (orders, separate service). Each story built on stable, tested foundations.

## Challenges & Struggles

### 1. Image Upload Security Complexity — 22 Review Patches

Story 5.3 had the most review patches of any story in the project. File upload/storage has disproportionate security surface: path traversal, content-type validation, TOCTOU races, orphaned file cleanup, concurrent upload guards. Initial implementation missed 9 security/race-condition issues.

### 2. PrimeVue + jsdom Incompatibility

PrimeVue 4.x components (Dialog, Drawer, AutoComplete) use teleport/portal APIs that don't resolve in jsdom. Every frontend story required workarounds:
- `shallowMount()` instead of `mount()` for Dialog-based components
- Custom stub factories for Drawer (teleports content outside wrapper)
- `vi.mock('vue-router')` timing conflicts with shared Pinia state

### 3. Pre-existing Test Environment Issue (ECJ Compiler)

VS Code Java Language Server (ECJ compiler) overwrites `ProductMapperImpl.class` with broken bytecode, causing integration test failures in Story 5.3. Tests were written correctly but could not execute. Pre-existing issue, not introduced by Epic 5.

### 4. No Defense-in-Depth on Admin Endpoints

All 3 admin controllers (`AdminProductRestController`, `InventoryAdminRestController`, `OrderAdminRestController`) rely solely on API Gateway for RBAC. No `@PreAuthorize` annotations. Direct access to any service bypasses auth. Risk accumulating — 3 services × multiple admin endpoints.

### 5. Concurrent Action Guards Missing in Every Story

4 out of 5 stories had review patches for missing concurrent action guards:
- 5.2: No `isDeleting` flag during bulk delete
- 5.3: Concurrent file-select without guard, drag-drop without in-flight check
- 5.4: Bulk restock button not disabled during request
- 5.5: Rapid double-click on status update dropdown

## Key Insights & Patterns

### Recurring Bug Patterns

1. **Nullable field handling (4/5 stories)**: `formatDate` crash with null, `shippingAddress` typed non-nullable, `description` hardcoded empty, `String.valueOf(null)` → `"null"` string
2. **Missing concurrent action guards (4/5 stories)**: Every interactive action needs loading state + disable guard
3. **PrimeVue component API differences**: Dropdown→Select, Sidebar→Drawer caught multiple times

### Technical Patterns Established

1. **Admin controller convention**: No @PreAuthorize, gateway RBAC, Tracer injection, OptimisticLock handler
2. **Pinia store template**: Composition API, ref() state, async actions, loading/error state
3. **Frontend enrichment pattern**: Client-side join via parallel API calls (inventory + products)
4. **File storage security checklist**: Path traversal check, magic bytes validation, TOCTOU serialization, orphan cleanup

### Process Insights

- **Pattern reuse drives velocity**: Once admin conventions were established in 5.1-5.2, stories 5.3-5.5 were significantly faster
- **File I/O stories need elevated security attention**: Treat as high-risk by default
- **Concurrent scenarios still missed in initial implementation**: Despite Epic 4 lesson, guards still added via review. Need process change (checklist).

## Key Takeaways

1. **File upload security needs dedicated attention** — 22 patches in one story. Path traversal, content-type spoofing, TOCTOU races. Treat file I/O as high-risk by default.
2. **Concurrent action guards must be a dev habit, not a review catch** — 4/5 stories had missing guards. Add to checklist to shift left.
3. **Pattern reuse is the velocity superpower** — Admin controller convention, Pinia store template enabled 5 stories in 3 days. Invest in establishing patterns early.
4. **Common-lib: forced closure after 5 epics** — Explicitly deferred to Epic 10. Stop the recurring discussion.
5. **3-layer code review continues to prove value** — CVE-level bugs caught before ship. Security is not optional for admin tools.
6. **Deferred work trending upward but manageable** — 75 total items. Epic 8 (Resilience) remains the critical remediation epic.
