## What Went Well

### 1. Redis-Only Microservice Architecture
The decision to build Cart Service as a Redis-only service (no JPA, no PostgreSQL) was a clean architectural choice. The `excludeName` pattern for auto-configuration exclusion worked well and established a precedent for future specialized services.

### 2. Event-Driven Cache Invalidation
Story 2.3 delivered a robust cache invalidation pattern via Kafka consumers. Separate consumer groups per service (product-service-cache-invalidation-group, cart-service-product.product.updated-group) ensured independent processing. This pattern is reusable for Epic 3+.

### 3. Optimistic UI with Rollback
The frontend optimistic update pattern (snapshot → update → API → sync/rollback) delivered excellent UX. Users see instant feedback while the system confirms server-side. The pattern was well-tested with explicit rollback verification.

### 4. Code Review Process Maturity
31 patches caught and applied across 4 stories — the 3-layer review (Blind Hunter, Edge Case Hunter, Acceptance Auditor) continued to prove effective. Key catches: integer overflow on quantity (2.1), unbounded memory leak in processedEventIds (2.3), missing per-item loading guards (2.4).

### 5. Testing Discipline
~198 tests with zero regressions. Backend used Testcontainers for Redis integration tests. Frontend adopted vi.mock() with flushPromises() pattern consistently. Test data builders continued from Epic 1.

## Challenges & Struggles

### 1. Jackson 3.x + Redis Serialization
GenericJackson2JsonRedisSerializer (Jackson 2.x) doesn't exist in Spring Boot 4 — had to use GenericJacksonJsonRedisSerializer from Jackson 3.x. Required polymorphic type validator configuration to prevent deserialization attacks while allowing project classes.

### 2. @Indexed Gotcha in Spring Data Redis
@Indexed on userId field created orphaned phantom keys in Redis that persisted after TTL expiry. Solution: use userId directly as cart.id instead. This was a non-obvious pitfall that cost debugging time.

### 3. Race Conditions (Systemic, Deferred)
Read-modify-write race conditions appeared in 3 out of 4 stories (2.1, 2.2, 2.3). Each story amplified the issue. The team correctly deferred to Epic 8 (resilience patterns with distributed locking), but the accumulation is concerning.

### 4. Spring Boot 4 Auto-configuration Discovery
Flyway autoconfigure moved out of spring-boot-autoconfigure into spring-boot-starter-flyway. CacheErrorHandler needed explicit configuration. These weren't documented in migration guides and required trial-and-error.

### 5. Full Cart Scan on Product Updates
ProductEventConsumer uses findAll() to scan every active cart on product price change. Acceptable for MVP but will not scale. No secondary index exists (productId → cartIds). Deferred to Epic 8.

## Key Insights & Patterns

### Technical Patterns Established

1. **@RedisHash + @TimeToLive**: Automatic TTL expiry with refresh-on-access pattern
2. **Spring Cache Abstraction**: @Cacheable/@CacheEvict with RedisCacheManager + CacheErrorHandler
3. **Kafka Consumer per Service**: Separate consumer groups for independent event processing
4. **Pinia Composition API**: defineStore with setup function (ref, computed, async actions)
5. **Optimistic UI**: Snapshot-based rollback for instant user feedback
6. **Axios Interceptors**: Header injection (X-User-Id) for anonymous cart tracking
7. **PrimeVue Toast with Actions**: Custom #message slot template for action buttons

### Process Insights

- **Review process scales well**: 31 patches across 4 stories with consistent quality
- **Deferred work tracking is critical**: 19 items documented prevents "invisible debt"
- **TTL-based caching simplifies invalidation**: 5-min product cache + event-driven eviction is a good balance
- **Pre-auth identity resolution works**: X-User-Id / X-Cart-Id header pattern bridges anonymous → authenticated (Epic 3)

### Architecture Decisions Validated

- Redis as primary store for ephemeral data (cart) — correct choice
- Spring Cache abstraction over raw Redis commands — right level of abstraction
- Kafka for cross-service cache invalidation — decoupled and reliable
- Optimistic UI over loading spinners — significantly better UX

## Key Takeaways

1. **Redis-only microservice is a valid pattern** — not every service needs JPA/PostgreSQL. Match storage to access patterns.
2. **Event-driven cache invalidation scales** — Kafka + separate consumer groups provide reliable, decoupled cache management.
3. **Optimistic UI significantly improves UX** — snapshot-based rollback is testable, predictable, and user-friendly.
4. **Code review remains the strongest quality gate** — 31 patches caught real issues (overflow, memory leaks, race conditions).
5. **Deferred work must be tracked explicitly** — 19 items would be invisible without structured documentation.
